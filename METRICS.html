Unlike traditional linters that simply count errors, Tech Debt Score uses a Density-Based Algorithm. This ensures that large monorepos are not unfairly penalized just for having more code.


The score is calculated as:

Score = 100 - (Total Penalty / Total Files) √ó Scale Factor

This measures the concentration of issues. A repository with 1,000 files and 1,000 issues will have the same score as a repository with 10 files and 10 issues.

Detects "God classes" and massive functions that violate the Single Responsibility Principle.

Files that do too much.

Threshold: > 300 lines

Function Length

Functions that should be split up.

Threshold: > 50 lines

Parameter Count

Functions with too many arguments.

Type Safety Rule Medium

Ensures TypeScript is being used effectively to prevent runtime errors.

Any Usage

Usage of the any type which bypasses type checking.

Threshold: > 0 usages

Code Duplication Rule Critical

Detects copy-pasted code using semantic token matching (ignores whitespace/renaming).

Duplicate Blocks

Identical code chunks appearing in multiple places.

Threshold: 50+ tokens, 95% similarity

Circular Dependency Rule Severe

Detects modules that tightly couple to each other, making refactoring impossible.

Module Cycles

Files importing each other (e.g., A -> B -> C -> A).

Threshold: Any cycle
